In the nlvm compiler at /data/projects/nlvm, crates/nl-codegen/src/expr.rs, the `peek_type` helper (around line 2751) used to fast-path string concatenation for `+` (in `compile_binary`, around line 2576) only recognizes `Expr::StringLit`, `Expr::IntLit`, `Expr::FloatLit`, `Expr::BoolLit`, `Expr::NullLit`, `Expr::Ident` (local var), and nested `Expr::Binary(BinOp::Add, ...)`. It does NOT recognize `Expr::FieldAccess` (e.g. `obj.field`) or method-call results. Consequence: an expression like `a.someStringField + b.someStringField` (two field accesses, both actually `string` at runtime, with no string literal or local-var operand anywhere in the `+` chain to "anchor" the peek) fails to take the string-concat fast path. It falls through to the generic numeric path, calls `promote_numeric(StringT, StringT)`, and hits the fallback arm at expr.rs:2706-2708, producing a hard codegen error: "unsupported construct: arithmetic/comparison between StringT and StringT" — even though the code is fully valid, well-typed NL (sema/type-checking accepts it fine; this is purely a codegen gap). Note also that nlc's error output for this class of `CodegenError::Unsupported` has no file/line location at all, just the bare message, which makes it hard to find in a larger program (see crates/nlc/src/main.rs error formatting and crates/nl-codegen/src/error.rs).

Reproduced live at /data/projects/nlvm.dev/src/Build.nl while writing an NL program: `string url = page.root + item.href;` (both `page` and `item` are plain fields on custom classes `PageSpec`/`NavItem` with `string` fields) failed to compile with exactly that error; it was worked around by copying each field into a local variable first (`string root = page.root; string href = item.href; url = root + href;`) so `peek_type`'s `Expr::Ident(local)` branch could recognize them.
